# クイズ書き起こし基準（じゅんばん・なかまわけ）

作成: 2026-07-04 ／ P6e改訂: 適用先は **tools/quiz-data.mjs の素材データ**（因果チェーン・タグ辞書）
P6e以降、じゅんばん・なかまわけも素材からのテンプレート生成に統一された。
このルールは「素材（チェーンの手順・アイテムのタグ）を人手で追加・修正するとき」に適用する。
一意性・難易度タグの機械検証は tools/quiz-criteria.mjs が正本（なかまわけの軸競合は全タグ軸照合で自動検出）。

## 作成ルール（全部満たさない案は捨てる）

1. **正解は常識で一意**。文化・家庭・地域で答えが変わる題材は使わない
   - NG例: 「おふろで さいしょに あらう ところ」「ごはんと おかず、どっちを さきに たべる？」
2. **誤答は絶対に成立しない**ものにする。もっともらしいが屁理屈で通る選択肢は禁止
   - 判定テスト: その選択肢に対して「〜という場合もあるよね」と言い訳できたら差し替え
   - NG例: 「がっこうから かえったら さいしょに することは？ → しゅくだい／おやつ／てあらい」（どれも成立する）
3. **じゅんばんは物理・因果の制約がある手順だけ**を使う（逆にすると不可能になるもの）
   - OK例: ふくを ぬぐ → おふろに はいる（着たまま入れない）／たまごを わる → まぜる
4. **なかまわけは分類軸を1つに固定**し、別の軸で括り直せないか必ず確認する
   - OK例: いぬ・ねこ・くるま（くるまだけ生き物でない。他の軸では括れない）
   - NG例: りんご・トマト・いちご（「たべもの」軸と「あかい」軸が競合し、なかまはずれが変わる）
5. **絵文字は意味が一目で確定するものだけ**。1問の中で同じ絵文字を使い回さない
6. **解説(why)は正解の理由を肯定形の1文で**（「ちがうから」ではなく「〜だからだね」）
7. 難易度の目安: やさしい=2ステップの手順・具体物の分類／ふつう=3〜4ステップ・機能による分類／
   むずかしい=抽象度の高い分類（かたち・ことばの種類）・「さいごに/2ばんめに」等の位置指定

## なかまわけの軸（PROP_AXES）を増やすときのルール（b4u 追記）

1. **qLabel の鉄則**: qLabel（出題文用の短い言い回し）は「その軸を持つ**全アイテム**に対して真」であること。
   どのアイテムが正解に抽選されても文が成立する言い方にする（例: water の qLabel「みずの なかや うえに いる」は さかな にも ふね にも真）。
2. **いろ/かたち軸（group つき concrete）の鉄則**: 絵文字は色・形が曖昧なものがある（たこ=赤っぽい・りんご=丸っぽい・にじ=何色でもある）。
   - prop は「色/形がはっきりした物だけ」に付ける。曖昧な物は**無タグのまま**にする。
   - 誤答は同グループ（color/shape）の**別 prop を持つ物だけ**から選ぶ（生成・検証の両方で機械担保）＝曖昧な無タグ物は色/形の問題に一切登場しない。
3. **「実は当てはまる」物には正直にタグを付ける**: 誤答側に回ると嘘になる物（うま/ぞう=ひとを のせて はこぶ・いす/ベッド=あしが4ほん）は、
   意外でもタグを付けて誤答事故を防ぐ。付けたうえで正解として自然かは --sample で目視確認する。
4. **★新軸を props に足すと仲間外れ形式の歩留まりが落ちる**（副作用・見落とすと問数が痩せる）:
   仲間外れ形式の検証（nakamaConflicts）は props の**全軸照合**なので、新しい prop は新しい競合を検出する（＝品質向上・歩留まり低下）。
   軸を足すときは **アイテムも一緒に増やし**、`node tools/quizgen.mjs --count` で各セルの到達を必ず数値確認してから量産する。
5. **安全パターン「同カテゴリ×軸だけ違い」**: はたらき軸の問題は なるべく同カテゴリ内で組む（例: のりものだけで「そらを とぶ」を切る）＝
   カテゴリで解けてしまう抜け道を塞ぎ、かつ他の軸との競合も起きにくい。同カテゴリ内で切れない軸（sound=がっきが全部鳴る等）だけカテゴリまたぎにする。

## レビュー手順

- 1問ごとに上記1〜6をセルフチェックし、1つでも引っかかったら修正でなく**破棄して作り直す**
- 全問作成後、カテゴリ×難易度の分布と、選択肢の重複・ID一意を `npm run verify`（構造チェック）にかける
