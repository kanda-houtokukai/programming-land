作成日: 2026-07-04 ／ 改訂: 2026-07-04 第2版（**これが正本**。第1版は本ファイルのgit履歴、v1はChatのアーティファクト）

第2版の改訂内容: 配布方式をGitHub Pages公開に変更、マルチプロファイル追加、育成ゲーム化（オリジナルモンスター）追加、フェーズ再編（P0〜P6）。

## 0. 前提と移植元

- リポジトリ内の `programming-land.jsx` が v1（動作確認済み・参照専用）。ゲームロジック・デザイン（ステッカー風UI・ひらがなUI・効果音）は原則そのまま移植する。
- v1棚卸し済み（2026-07-04）。保存APIの使用箇所は3か所のみ、効果音は自動再生制限対策済み、データはすべて純データとして分離可能であることを確認済み。

## 1. 技術方針（決定事項）

- Vite + React の静的ビルド。
- [DECISION 2026-07-04改訂] 配布は **GitHub Pages 公開（無料）**。タブレットのブラウザからURLで起動できること。単一HTML配布（vite-plugin-singlefile）は廃止。
  - アカウント: kanda-houtokukai（自分側・報酬改定ナビ等で実績あり）
  - デプロイ: `npm run deploy` 一発（ローカルで verify→build→docs/ にコピー→commit→push、mainブランチの docs/ フォルダから配信）。push後1〜2分で反映、確認はシークレットウィンドウで（キャッシュ回避）。
    ※当初は GitHub Actions 方式の計画だったが、保存済みトークンに workflow 権限がなくワークフローファイルを push できないため、報酬改定ナビと同じブランチ配信方式に変更（2026-07-04）。検証(verify)は deploy コマンドに組み込み、FAILだとpushされない。
  - **アプリ内にバージョン表示を付ける**（反映確認用の既存慣習）。
- ネット接続はページの読み込みのみ。サーバ・DB・外部アカウント・課金は作らない。
- 保存: localStorage（プロファイル別）。「きろくの かきだし／よみこみ」（JSONダウンロード・ファイル選択で復元）を保護者画面に必ず実装し、**プロファイル単位**で書き出し・読み込みできること（端末間の引き継ぎ・故障対策）。
- データ駆動: ステージ・クイズ・タイピング課題・モンスター定義はすべてデータファイル（ts/json）に分離。本体コードに問題をハードコードしない。

## 2. マルチプロファイル（決定事項 2026-07-04）

- 起動時に「だれが あそぶ？」選択画面（**最大4人**、ニックネーム＋アバター）。
- 記録・相棒・バッジは人ごとに**完全分離**して localStorage に保存。
- 書き出し／読み込みもプロファイル単位。

## 3. 育成ゲーム化（決定事項 2026-07-04）

- **オリジナルモンスター**。実在IP（キャラクター素材・名前）の流用は禁止。デザインは絵文字と自作SVGのみ。
- はじめに相棒を**3体から1体えらぶ**（タイプ違いのオリジナルデザイン、SVG自作）。
- パズル・クイズ・タイピングのクリアで**経験値→レベルアップ→3段階進化**。
- ワールド選択を**「島マップ」画面**に変更し、道をたどって進む見た目にする（難易度＝島のちがい、クリアで道が伸びる）。
- **ずかん**: 実績達成で仲間モンスターが増えるコレクション要素。
- **クイズバトル**: 正解すると技が出るターン制・対CPU（フェーズP6）。

## 4. 難易度システム（全モード共通・決定事項）

- 3段階: ⭐やさしい ／ ⭐⭐ふつう ／ ⭐⭐⭐むずかしい。モード入口で選択、いつでも変更可。
- 記録は難易度別に保存・集計（保護者画面に難易度別の到達度を表示）。
- 解放条件は難易度内で完結させる（むずかしいを遊ぶのに、やさしいの全クリアは要求しない）。

## 5. フェーズ計画と完了条件（第2版で再編・P0範囲の承認をもって確定）

フェーズ番号は第1版から変わったため、以後は必ず番号＋名前で呼ぶ（例:「P2（パズル増量）」）。

### P0: きばん（v1移植＋公開＋プロファイル）
- Vite + React 骨組み、GitHub リポジトリ公開、GitHub Pages 自動デプロイ、バージョン表示。
- v1の3モード＋きろく画面を移植。データ4種（ステージ・クイズ・おえかき・バッジ）をデータファイルに分離。
- マルチプロファイル（§2）を実装。v1のはじめて画面はプロファイル作成画面に発展させる。
- localStorage保存＋プロファイル単位の書き出し/読み込み。セーブスキーマは相棒・経験値・ずかん・難易度別記録の項目を**最初から予約**しておく（後のフェーズでスキーマ変更を繰り返さない）。
- v1棚卸しで見つけた修正を反映: ①日付がUTC基準で朝9時前のプレイが前日扱いになるバグ→日本時間基準に修正 ②「星45個」等のハードコード→ステージ数からの自動計算に変更。Googleフォントは現状維持（オフライン時はOS内蔵の丸ゴシックに自動フォールバック）。
- ソルバー＋`npm run verify` を作成し、既存15ステージを全数検証（★3解の存在・parが最短解と一致）。
- 完了条件: **公開URLでPC（Safari含む）・タブレットの実機確認**、verify全PASS、書き出したJSONの読み込みで完全復元。

### P1: そだてる基盤（相棒システム）
- 相棒モンスター3体×3段階進化＝9姿のSVGを自作。タイプ違いのオリジナルデザイン。
- 経験値・レベル・進化の仕組み。パズル・クイズ・おえかきのクリアから経験値付与（タイピングはP4で合流）。
- ずかんの骨組み（相棒と進化形の登録・閲覧）。
- 完了条件: 相棒選択→遊んで経験値→レベルアップ→進化演出までが動作し、記録画面に相棒状態が表示される。

### P2: パズル大増量＋島マップ（150ステージ以上）
- ワールド6つ＝**6つの島**に拡張: ①じゅんばん ②くりかえし ③もしも ④くみあわせ（②＋③混合）⑤あたまのたいそう（最短ブロック競争）⑥ちょうせんじょう（大型迷路）。
- 各島×3難易度×8〜10ステージ ≒ 150〜180。
- 島マップ画面: 道をたどって進む見た目、クリアで道が伸びる、難易度＝島のちがい。
- **生成スクリプト＋自動検証を必ず作る**: ステージはジェネレータで量産し、ソルバーで「★3解が存在」「★3のpar値が最短解と一致」「意図した難易度レンジ（解の長さ・使用ブロック種）」を全数チェック。検証を通らないステージは採用しない。手作業で150面作らないこと。
- 完了条件: `npm run verify` で全ステージ PASS がCI的に確認できる。

### P3: クイズ大増量（200問以上）
- 既存4カテゴリ＋「よみとり（フローチャートの絵を読む）」を追加し5カテゴリ×3難易度。
- きまり発見・ロボット命令系は**テンプレート生成**（絵文字パターンや歩数を機械生成し正答も機械算出）で量産可。じゅんばん・なかまわけ系は固定問題を書き起こす。誤答が成立しない問題（正解が複数ある等）を防ぐ生成ルールを設ける。
- 完了条件: 総問題数200以上、各問題に解説文、難易度タグ付与、出題はセット内シャッフル。

### P4: タイピングモード（新規）
- 小1〜2向け3段階:
  1. やさしい「キーを さがせ」— 画面のキーボード図で光ったキーを押す（ホームポジション導入、指の色分け）
  2. ふつう「ひらがな ことば」— 2〜4文字のことばをローマ字入力（画面に押すキーのガイド常時表示）
  3. むずかしい「ぶんしょう」— 短文、ガイドは薄く
- ローマ字は訓令式・ヘボン式の両方を正解にする（si/shi, ti/chi 等）。**変換テーブルの単体テストを書く**。
- 記録: レベル別ベスト（正確率・1分あたり文字数）、練習した日を既存ログに合流。タイピングのクリアも経験値に合流。
- 完了条件: 外付けキーボードで全レベルが遊べ、テスト PASS。

### P5: きろく・バッジ・ずかん拡張
- バッジを難易度・タイピング対応で25個程度に増補。ずかんの仲間モンスターを実績達成で増やす。
- 保護者画面: **プロファイル別表示**、難易度別到達度、「キーボード入力」項目を追加。学習内容の対応表（順次・反復・分岐・タイピング）は維持する。
- 完了条件: 書き出したJSONを読み込むと完全復元できる。

### P6: クイズバトル
- 正解すると技が出るターン制バトル・対CPU。相棒モンスターが戦う。
- 完了条件: 3難易度のCPU相手に1戦が最後まで遊べ、勝敗が記録・経験値に反映される。

- v1のUIトーン（ひらがな・ステッカー風・効果音）の全面刷新

## 7. 検証と落とし穴
- 各フェーズ完了時に必ずビルド→**公開URLで実機確認（PC Safari＋タブレット）**。プレビュー成功≠完成。Pages反映はpush後1〜2分、確認はシークレットウィンドウで。
- dev サーバ稼働中に `npm run build` を実行しない（ビルド破損の既知事故と同型）。
- 効果音は AudioContext をユーザー操作後に初期化（自動再生制限対策。v1実装を踏襲）。
- 保存データはバージョン番号を持たせ、将来の項目追加でクラッシュしない読み込み（デフォルト値マージ）にする。v1実装の `{...newSave(), ...parsed}` 方式を踏襲。データ形式を変える改修では必ずマイグレーション（自動変換）を同梱する（勤務表ツールの教訓）。
- 保存は端末内のみ。タブレットとPCで記録は共有されない旨を保護者画面に明記し、引き継ぎは書き出し/読み込みで行う。
