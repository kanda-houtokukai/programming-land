## §1. Chat 側の調査結果（そのまま信じず、§2 で再現を取ってから着手する）

対象: `src/components/WorkshopEditor.jsx`

1. パレット由来のドラッグが「無言で」消える経路は1本しかない。 `pointercancel` → `onCancel` → `dropLastSnapshot()`。何も置かず、演出も音もトーストも出ない。 一方作業エリアの外で離した場合は派手（ポン演出＋`sndPoof`＋トースト「とりけし」）。神田さんが見ているのは前者。
2. 棚のカードに `touch-action: pan-y`（`.studio-pal .pal`）。 指のジェスチャは Safari が縦スクロールとして横取りでき、取られた瞬間に `pointercancel` が飛ぶ。 `preventDefault()` も `setPointerCapture` もこれを止められない（`touch-action` の方が強い）。 マウスには `touch-action` が適用されない ＝ 「指だけ失敗する」の説明が付く。
3. 「きっかけだけ成功しやすい」の説明。 棚の最上段はスクロールの余地が乏しく Safari が横取りしない。中ほどのカードは上下に余地があるため取られる。
4. 指とマウスを区別する処理が皆無。 `pointerType` の参照 0箇所、`touchstart` / `touchmove` リスナ 0箇所。指はマウスと同じしきい値で扱われている。
5. もう1つの無言失敗経路。 `CFG.PAL_PRE_MOVE = 7`（px）／`CFG.PAL_LONGPRESS = 150`（ms）。 150ms のあいだ 7px 以内で静止できないと、ながおしが成立せず不発。指は接地面のにじみで動くため、マウス基準のこの値は指にはきつい。低学年の指ならなおさら。

＝ 塞ぐべき穴は2つある。片方だけでは直らない。

## §3. 直す方針（★方向で分ける）

dev-workflow スキル §5 の落とし穴に、この案件の確立済み作法として **「ドラッグとスクロールは方向で分ける（`pan-y` ＋ 移動方向の判定）」** が既に載っている。現行は `pan-y` は入れたが判定が時間ベース（150ms 静止）で、方向判定が入っていない ＝ 作法の半分だけ適用された状態。これを揃える。

### 3-1. よこに動いたら即ドラッグ / たてに動いたらスクロール

* 棚のカードの上で横方向の移動が縦方向を明確に上回ったら、その場でドラッグを開始する（静止待ちを要求しない）。
* 縦方向が上回ったらブラウザのスクロールに譲る（現行どおり）。
* 判定は最初の数pxで確定させ、一度決めたらそのジェスチャ中は切り替えない（方向ロック）。

### 3-2. ながおし → せつめいふきだし は残す

落とし穴「触れただけで確定するUIにしない ＝ ながおし→説明→ドラッグの三段導線」は教育上の設計なので壊さない。 指を置いたまま動かさなければ 150ms でせつめいが出る、横に動かせばドラッグが始まる、の両立にする。

### 3-3. つかんだ後の保険

ドラッグ成立中は `touchmove` を非パッシブで受けて `preventDefault()` し、Safari の横取りを止める。 ⚠️ ドラッグ成立中だけにすること。常時かけると棚の縦スクロールが指で死ぬ（落とし穴に記載の過去失敗と同じ形）。

### 3-4. しきい値

* `CFG.PAL_PRE_MOVE = 7` はマウス基準。方向判定を入れるなら、この「静止を要求する」門自体が不要になる可能性が高い。残すか外すかは Code が判断してよいが、判断と理由を報告すること。
* 指と マウスで値を変えるなら `e.pointerType` で分岐する（現在0箇所なので新規導入）。
