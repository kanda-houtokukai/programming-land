## §0. スコープ（2点）

1. **白いチップがメスのくぼみに干渉**：カード内の白い四角（アイコン台）の上端が、上辺のメスのくぼみ（`TD:8`）に 1px 食い込む（容器＝くりかえし/ずっとは 3px でより目立つ）。
2. **こうぐだなが指で縦スクロールできない**：`.studio-pal` に `overflow-y:auto` と `touch-action:none` が同居し、縦スワイプが殺されている（下のカードに届かない）。タッチ専用の不具合。

**やらないこと**：ブロックの形・凸凹（パス）は変えない。パレットの枠/見出しの整理・自動幅・背景変更は**ここではやらない**（gamelab 段階2 の作り直しで扱う）。studio の見た目はこの2点以外そのまま。

## §1. 修正A：チップとくぼみの干渉（`src/workshop/geometry.js` ＋ `StudioBlock`）

原因：チップは上下中央配置だが、メスのくぼみ底が y=8。通常ブロックのチップ上端は y=7、容器は y=5 で食い込む。

対応（前回モックで合意した値）:
1. `geometry.js` の `G`：**`CHIP: 34 → 30`**、**`ICON: 26 → 24`**。
   - これで通常ブロックの中央チップ上端は `(48-30)/2 = 9` となり、**くぼみ底(8)を自然に避けつつ中央のまま**（むしろブロック中心と一致）。
2. **チップ上端を y≥9 にクランプ**（容器対策）。チップY算出（`StudioBlock` の `chipY` 相当）を:
   ```js
   const cy = Math.max(chipY(shape), 9);   // メスのくぼみ底 TD=8 を避ける
   ```
   - 通常= max(9,9)=9（変化なし）／容器= max((44-30)/2=7, 9)=9（2px下げ）／はた= max(14,9)=14（変化なし・くぼみなし）。
   - アイコンのオフセットも `(CHIP-ICON)/2 = 3` で中央（`iconOff` を CHIP/ICON から算出しているなら自動）。
3. **文字（ラベル）は動かさない**（`labelY` はそのまま）。

**★ベースライン再取得（必須）**：`CHIP/ICON/chipY` は凍結ジオメトリの一部なので、この変更で `tools/studio-baseline.json` の**チップ関連の計測値が変わる**。
- 対応：チップ修正の**後に** `test-studio-regression.mjs --update` で**ベースラインを取り直す**。
- **検証（重要）**：再取得の diff を必ず確認し、**変わったのがチップ関連（chip 位置/サイズ・measures のチップ項）だけ**であること。**パス（pathBody/Hat/C）・幅(widths)・エンジントレース(732イベント)は1バイトも変わらないはず**。もしパス・幅・トレースが変わっていたら**赤信号＝止めて原因究明**（チップはブロックの外形パスに影響しないため変わってはいけない）。
- gamelab も同じ `StudioBlock`/geometry を使うので同時に直る。

実機確認：通常ブロックと**特に容器（くりかえし・ずっと）**で、白いチップがくぼみに触れていないこと。アイコンが小さくなり過ぎていないこと（24pxで従来とほぼ同じ）。

## §2. 修正B：こうぐだなのタッチスクロール（`src/components/WorkshopEditor.jsx`）

原因：`.studio-pal`（スクロール容器）の `touch-action:none` が縦スワイプを無効化。さらに各カードは `onPalPointerDown` で即 `preventDefault()` してドラッグ開始するため、カードの上でもスクロールに回らない。

**レイアウトが味方**：こうぐだなは左・組み立ては右なので、**カード取り出し＝横方向／スクロール＝縦方向**に分かれる。これを使う。

対応:
1. CSS：`.studio-pal` と `.studio-pal .pal` の **`touch-action: none` → `touch-action: pan-y`**（縦スワイプはブラウザにスクロールさせる。横方向はアプリが取れる）。`.studio-pal` の `overflow-y:auto` はそのまま。
2. `onPalPointerDown`（パレットカードの pointerdown）を**即ドラッグ開始しない**形に:
   - pointerdown では `preventDefault()` を**呼ばず**、開始座標・pending 状態だけ記録。
   - 最初の `pointermove` で判定：
     - **横向きに閾値（例 6〜8px）を超えたら**→ そこで `preventDefault()` ＋ ドラッグ開始（従来の取り出し処理へ）。
     - **縦向きが優勢**→ 何もしない（`pan-y` によりブラウザがスクロール）。ブラウザがスクロールを取ると `pointercancel` が来るので、pending ドラッグを破棄。
   - `pointercancel` ハンドラ：pending / 進行中の pull-out を中断。
3. `.studio-asm`（組み立て）・`.theater`（上演）の `touch-action:none` は**そのまま**（あそこはドラッグ/タップ用で正しい）。修正はパレットだけ。

検証（iPad 実機）：
- こうぐだなを**指で縦スワイプ→下のカードまでスクロールできる**（gamelab の20種で下のカズまで届く）。
- カードを**横に引く→ブロックが取り出せる**（従来どおり）。
- studio・gamelab **両方**で成立。組み立て/上演のドラッグ・タップは不変。

ベースライン影響：なし（CSS/イベントのみ・ジオメトリ非関与）。回帰ハーネスは修正Aの再取得後 GREEN であること。

## §4. 落とし穴

- チップ修正の後は**必ずベースライン再取得**し、diff が**チップ関連だけ**か確認（パス/幅/トレースが動いたら止める）。
- `touch-action` は**パレット容器＋カードだけ** `pan-y` に。組み立て/上演は `none` のまま。
- ドラッグ開始を pointerdown 即時から**横移動しきい値**に変える点を忘れない（でないとスクロールに回らない）。
- dev サーバ稼働中に build/deploy しない。
