## 作業（1パス・通し・小さく積む順）

### 1. なかまわけの軸を増やす（③の核心）
- **`PROP_AXES` に `type:"concrete"` の新軸（いろ・かたち）を追加**。criteria の `expectedDifficulty` は**既に `concrete→easy` を持つ**ので、難易度マッピングの変更は不要。
- `genNakamaAxis()` の **easy 分岐を「カテゴリ軸 or 概念軸(concrete prop)」から選べるよう拡張**（既存のカテゴリ軸経路は残す）。
- functional / abstract の軸も追加（現在 functional 5・abstract 3 しかなく、hard「たべられる」が5問に膨らむ偏りの原因）。
- **qLabel の鉄則**: qLabel は「その軸を持つ**全アイテム**に対して真」であること。
- **いろ軸の鉄則**: 絵文字は色が曖昧なものがある（車など）。**色がはっきりした物だけ**に色 prop を付ける。

### 2. ★副作用を監視しながらアイテムを増やす（見落とすと痩せる）
仲間外れ形式の検証は `nakamaConflicts()` ＝ **props を使った全軸照合**。**色・かたちを props に足すと、従来の仲間外れ形式が新たな競合を検出して弾かれ、収束しにくくなる／問数が痩せる。**
- これは**品質向上**（潜在的あいまいさを捕まえる）だが**歩留まりは落ちる**。
- 対策: **軸追加と同時に `NAKAMA_ITEMS`（現在70）も増やす** → `--count` で歩留まりを確認 → 痩せすぎなら軸の付け方（どのアイテムに付けるか）を見直す。

### 3. 各カテゴリの素材追加（設計 doc §4）
- **じゅんばん最優先**: 異なる話が easy14/normal9/hard14 と少ない（normal 9 は R3 の余裕が薄い）。**因果チェーン＝話数を増やす**。
- きまり: 絵文字プール追加（今10→+10目安）。よみとり: seq/branch/loop/loop2 素材追加（hard は丸かぶり64%＝答えの語彙が少ない）。ロボット: 手続き生成のため優先度低。
- **1カテゴリ足すごとに `--count` で到達を確認**。

### 4. 【関所A】--count で数値化・報告して続行
- 各セルが40〜50に届くかを報告。届かないセルは最低30まで下げて続行。

### 5. N を確定して再生成
- `COUNTS`（`nakama:32` 起点）を --count で確定した値に更新 → `node tools/quizgen.mjs --write` で再生成。
- **生成後、実データを目視検分**（新軸の問題文が自然か、色の曖昧な組が無いか）。

### 6. ①② 出題ロジック改修（`src/data/quizzes.js` の `buildSession`）
**b4t の現行360問のままで全15セル100%充足を実測確認済み。** 任意の2問が次に該当しないよう5問を選ぶ:
- **R1**: 問題文が同じ **かつ** 正解ラベルが同じ（junban/yomitori の実質同一問題を排除。※文言だけの一致は kimari・仲間外れでは正常なので**禁止しない**）
- **R2**: 選択肢セット（ソート済み）が同じ（丸かぶり排除）
- **R3**: junban: `meta.chainId` が同じ（同じ話を2回出さない）
- **R4**: nakama: `meta.items` のラベルを**2つ以上共有**
- **R5**: nakama-axis 同士: `meta.axis` が同じ（＝同じ問題文になる）
- 満たせないときのみ R1・R2 だけに緩め、最後はレコード非重複のみ。**必ず5問そろえる**。
- **②**: 選んだ5問の正解位置を選択肢数ごとに均等配分（3択=2:2:1／4択=2:1:1:1）して割り当て、正解が目標位置に来るよう選択肢を並べ替える。**nakama/hard は3択と4択が混在**するので選択肢数ごとに配分すること。
- `battlePool` は壊さない（②の位置均等はバトルに入れてもよい・任意）。

### 7. ドキュメント・UIコピー整合
