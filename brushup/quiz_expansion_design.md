## 0-1. 実測診断（b4t 360問で取り直し・各セル3万セッション）

- **「5問すべて同じ位置」＝約1.2〜1.3%**（3択）。正解位置の分布は均等でシャッフルは正常。**バグではなく確率どおりの偶然**（「全部いちばん上」は約0.4%）。→ ②で構造的に根絶する。
- **「選択肢の丸かぶり」は依然として多い**: yomitori/hard **64.4%**、robot/normal **50.6%**、junban/easy 34.0%、yomitori/easy 30.7%、robot/hard 31.6%、kimari/easy 23.0%。
- **★b4t の新しい副作用**: 軸名称形式は**同じ軸＝同じ問題文**になる。hard「たべられる なかまは どれ？」が**5問**、normal「そらを とぶ」3問・「ものを きる」3問、easy「がっきの なかま」2問。2問が同時に出ると**文言まで同一**で"同じ問題"に見える。
- **★実例で判明（①の設計を変えた根拠）**: 「同じ問題文」の中身はカテゴリで意味が違う。
  - junban/normal「ホットケーキを つくるよ。さいしょに することは？」×3 → **同じ話・同じ設問・同じ正解**で誤答だけ違う＝**実質同一問題**。
  - yomitori/normal も同様（同じフローチャート・同じ設問・同じ正解）。
  - 一方 kimari/hard「きまりが ちがうのは どれ？」×8 と 仲間外れ「なかまはずれは どれ？」×24 は**汎用の問いかけ**で中身は選択肢側＝**別問題**（文言一致は正常）。
  → **文言の一致だけでは判定できない。「文言＋正解」が一致したら実質同一**、という鍵が要る。

## 2. ①②の仕様（純ロジック・データ再生成なし・**b4t 360問のままで100%充足を実測確認済み**）

`src/data/quizzes.js` の `buildSession(category, difficulty)` を2段構えにする。

### 2-1. 5問を選ぶ（①）— 実測で確定したルール R1〜R5
プールから、任意の2問が次のいずれにも該当しないように5問を選ぶ:

| | ルール | 目的（実測根拠） |
|---|---|---|
| **R1** | **問題文が同じ かつ 正解ラベルが同じ** | junban/yomitori の「実質同一問題」を排除 |
| **R2** | **選択肢セット（ソート済み）が同じ** | 丸かぶり（yomitori/hard 64%・robot/normal 51%）を排除 |
| **R3** | junban: **`meta.chainId` が同じ** | 同じ話を1セッションに2回出さない（異なる話は easy14/normal9/hard14 あり5話確保可） |
| **R4** | nakama: **アイテムを2つ以上共有**（`meta.items` のラベル基準） | 仲間外れ形式の"似た問題"を排除 |
| **R5** | nakama-axis 同士: **`meta.axis` が同じ** | 同じ軸＝同じ問題文（hard「たべられる」×5）を排除 |

- **満たせないときのみ**、R1・R2 だけに緩める（最後はレコード非重複のみ）。**必ず5問そろえる**。
- 実測: b4t の現行360問で **全15セルが100%充足**（各5000回）。フォールバックは保険として残すだけ。
- 在庫を増やす③のあとは制約がさらに緩くなるので、充足性は悪化しない。

### 2-2. 正解位置を割り振る（②）
選んだ5問の正解位置を、各問の選択肢数に応じ**均等な多重集合**にしてから割り当てる（3択=2:2:1／4択=2:1:1:1）。順番はシャッフルし、各問は正解が目標位置に来るよう選択肢を並べ替える（見せ順は自由なので正誤に影響しない）。**「全問おなじ位置」は構造的に発生しない。**
※ nakama/hard は3択と4択が混在する（軸名称hardが4択）。選択肢数ごとに配分すること。

### 2-3. バトルへの波及
`battlePool`（`Battle.jsx` と共用）は独自のシャッフル済みキューから1問ずつ引くため直近重複は自然に避かる。**R1〜R5 はクイズの `buildSession` 側のみ**。②の位置均等はバトルに入れてもよい（任意）。最低限**`battlePool` を壊さない**。

## 3. ③ なかまわけ＝**拡張**（※「新設」は誤り・b4t現物に合わせ全面改訂）

### 3-1. 何が既にあり、何を足すのか
既にある: 軸名称形式（さがす言い方「〇〇の なかまは どれ？」）／カテゴリ軸(easy)・はたらき軸(normal)・抽象軸(hard)／criteria の一意性検証。
**足すのは「軸の種類」と「言い方」と「比率」の3つ。**

### 3-2. 足す軸（`PROP_AXES` は現在8軸しかない＝偏りの原因）
現状 hard は abstract 3軸しかなく「たべられる」が5問に膨らんでいる。**軸を増やすことが、文言重複・偏りへの直接の対策**になる。

- **★concrete 型の prop 軸を新設（いろ・かたち）** … criteria の `expectedDifficulty` は **既に `concrete→easy` を持っている**ので、`PROP_AXES` に `type:"concrete"` を足し、`genNakamaAxis` の easy 分岐で「カテゴリ軸 or 概念軸」を選べるようにすれば**判定はそのまま効く**（criteria 側の難易度マッピング変更は不要）。
- functional の追加候補: すみか（みず/そら/りく系。water は既存なので重複させない）・のる/はこぶ・きる以外の道具のはたらき 等
- abstract の追加候補: いきているか以外の抽象軸（例: ひとが つくった／かたち が まるい 等は concrete 寄り。要吟味）

**qLabel の鉄則（b4t のコメントに明記されている）**: qLabel は「**その軸を持つ全アイテムに対して真**」であること（どれが正解に選ばれてもよいように）。
**いろ軸の鉄則**: 絵文字は色が曖昧なものがある（車など）。**色がはっきりした物だけ**に色 prop を付ける。

### 3-3. ★新軸を props に足すことの副作用（最重要・見落とすと痩せる）
仲間外れ形式(kind="nakama")の検証は `nakamaConflicts(m.items, m.odd)` ＝**props を使った全軸照合**で「別の軸でも割れる組」を弾いている。
→ **色・かたちを props に足すと、従来の仲間外れ形式が新たに競合を検出して弾かれ、収束しにくくなる／問数が痩せる**。

これは**バグではなく品質の向上**（りんご/トマト/いちご問題を色でも検出できるようになる＝今は見逃している潜在的あいまいさを捕まえる）。ただし**歩留まりは落ちる**ので、対策は:
1. **軸を足すと同時にアイテムも増やす**（70→増やせば有効な組が増える）
2. **`--count` で歩留まりを必ず数値確認**してから量産（想定外に痩せたら軸の付け方を見直す）
3. 比率調整（下記 3-4）

### 3-4. 比率（NAKAMA_ODD と COUNTS.nakama）
現在 32問中 仲間外れ24・軸名称8。軸を増やすなら**軸名称の比率を上げる**のが自然（軸名称は一意性が明示的に検証され安全・かつ増やしやすい）。
- **★COUNTS は `nakama:32` を起点に（24 に戻さない）**（本線申し送り）
- NAKAMA_ODD は「仲間外れの問数」。値の変更は可だが**機構は壊さない**。変更したら報告する。

### 3-5. 「のけもの」言い方（任意・余力があれば）
「**あかく ないのは どれ？**」＝1つだけ当てはまらない形式。既存 nakama-axis の検証は「当てはまるのが**ちょうど1つ**」を要求するため、**そのまま流用すると必ず落ちる**。
→ 追加するなら **`kind:"nakama-axis-neg"` として新設**し、既存 `nakama-axis` の検証は**一切変えず**、neg 用に「当てはまらないのがちょうど1つ（＝matches が n-1）」を検証する分岐を足す。優先度は低（軸追加が先）。
