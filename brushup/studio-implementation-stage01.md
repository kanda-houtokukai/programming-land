# 実装指示書: つくるスタジオ 段階0+段階1

作成: 2026-07-17 ／ 対象: Claude Code ／ 設計の正本: `studio-design.md`
**手触りの正本: `studio-block-prototype.html` 第11版** — 本指示書は「この見た目と挙動を
数値ごと1:1でReactに移植せよ」という指示である。解釈で変えないこと。

推奨モデル: **Fable 5**(ドラッグ&スナップ系+並行インタプリタの新規エンジンで、
本案件最高難度。dev-workflowの基準に該当)。段階0完了後の段階1続行は同セッションで可。

---

## 0. 前提とルール

- 既存機能・既存ファイルの挙動は一切変えない。ワールドマップにも**まだ触らない**
  (マップ設置は段階3)。アクセスは開発用の一時ルートで行う(§2)。
- 絵文字アイコン禁止。**生成イラスト18種は完成済み**(`studio-assets.zip`)。段階0から実物を使う
  ——プロトタイプ第11版と同じく、**白チップ(#FFFDF6・34px・rx11)の上にアイコンを載せる**
  (直置きは18個中13個がカード色に埋没するため禁止。設計書§10参照)。
  効果音のみ段階1はWebAudio簡易音(プロトタイプの音をそのまま移植)でよい(Suno音源差し替えは段階3)。
- **matchMedia / ResizeObserver 禁止**(過去2件の障害実績)。縮尺の再計算は
  window resize イベント+向きに依存しない同一式で行う(プロトタイプ準拠)。
- verify全PASS・`src/version.js`更新・documents-onlyコミットは手動push、は従来どおり。
- 調整値(TICK・磁石3値・上限類)は必ず設定オブジェクトに集約し、マジックナンバーで
  散らさない(憲章§4-4)。

## 2. 段階0: ブロック見本画面(見た目ゲート)

**目的**: 機能を積む前に、ブロックの見た目と手触りだけを実機合格させる。

作るもの: プロトタイプの機能範囲(パレット/掴むと下が束で外れる/磁石スナップ/
救済スナップ/入れ子深さ2/はた1本制限/ステッパー/おと切替/たなへ捨てる/効果音)を
Reactで再現した単独画面。上演ステージは**含めない**(段階1)。

- アクセス: URLハッシュ `#studio-dev` でのみ表示される一時ルート(マップ導線なし)。
- 移植の要点:
  - ジオメトリ定数・パス生成関数・磁石3値(78/1.3/140)・アニメ数値(隙間180ms
    cubic-bezier(.25,.9,.35,1.25)/吸着110ms/掴み2.6deg,1.05/着地200ms)は
    プロトタイプの値を**変更禁止**。studio-design.md §11が数値の正本。
  - DOM戦略もプロトタイプ準拠: ブロック=絶対配置div+SVGパス、位置はtransform、
    隙間の開閉はレイアウト再計算+CSS transitionで表現。ドラッグはPointer Events
    直書き(D&Dライブラリ不使用)。`touch-action:none`・最初の1本指のみ・
    user-select無効・初回pointerdownでAudioContext解錠。
  - ピル文字色=そのブロックの縁系濃色(白地に白文字の事故防止)。
  - アイコンは `<image href=...>` でチップ上に配置。チップy = 帽子:14 / 容器:(TB-CHIP)/2 /
    通常:(H-CHIP)/2。ラベル開始x=46(LABELX)。ブロック幅はチップ分+10px済みの値を使う。
- 【段階0完了で停止】神田さんがiPad実機で、プロトタイプと同じ手触りかを合格判定。
  不合格項目は数値調整で追い込む(機能追加はしない)。

## 3. 段階1: 3ペインエディタ+実行エンジン

### 3-1. レイアウト
- 左: こうぐだな(カテゴリ色の縦パレット・全18種)。
- 中央: 組み立てエリア(選択中キャラのプログラムのみ表示)。
- 右: ステージペイン `clamp(280px, 36vw, 620px)`・縦横比3:2。**論理12×8マス固定・
  常に全体を縮尺表示**。+「ひろげる」全画面上演トグル(編集UI消灯・■で復帰)。
- 幅が不足する画面(スマホ)は「タブレットか パソコンで あそんでね」の案内のみ表示。
  判定はCSS幅のみ。

### 3-2. ステージとキャラ
- 背景セレクタ: だいち/ジャングル/キャニオン/アリーナ/無地スタジオ(既存WebP流用、
  IDの配列で管理し追加可能に)。
- キャラ最大5体・同種重複可。控え室(はいゆうひかえしつ)から追加:
  顔ぶれ=主人公+ずかんの出会った相棒+倒した敵(最低保証: 主人公+初期相棒+敵2体)。
- 配置はステージ上ドラッグ(見えない格子スナップ)。編集時位置=初期位置。
  重なり順=後置きが手前固定。初期大きさ1倍。選択中キャラは足元スポットライト。
- キャラ削除=控え室へドラッグ。確認1回「この こと プログラムを けす?——とりけしで
  もどせるよ」。

### 3-3. ブロックとプログラム
- 全18種(studio-design.md §5の表が仕様)。きっかけ毎1本/スタック上限30/浮きスタック合法/
  入れ子深さ2/ずっとは末尾のみ、を形状とスナップ候補で強制。
- とりけし/やりなおし: 履歴20手(ブロック木+キャラ配置のスナップショット)。
  ブロック増減・移動・数値・キャラ増減すべて対象。
- コピー: ブロック長押し0.5秒(6px動いたらドラッグ判定)→「コピー」バルーン→複製が
  隣に出現しそのまま掴める。

### 3-4. 実行エンジン(engine.js)
- TICK=400ms。▶(カチンコ音)で全キャラの「はた」スタックを並行開始。
- 同拍競合の処理順: はた→タップ→ぶつかり。
- 各カード挙動はstudio-design.md §5の表に厳密準拠。特に:
  端で残り歩数破棄/きえていても走り続ける(タップ・ぶつかり対象外)/
  ぶつかりは重なった瞬間のみ・離れて再接触で再発火/もとのばしょへは位置のみ。
- 実行中: 実行ブロック発光(入れ子内も)・たな/組立エリア暗転+操作ロック。
- ■=カット!: 全停止+全キャラ初期化(位置/大きさ/表示)。自然終了は位置維持・
  次の▶で初期化してから開始。
- エンジンはUI非依存(状態変化をコールバックで通知)にし、単体テスト可能な形に。
  最低限のテスト: 端で歩数破棄/入れ子くりかえしの回数/ずっとの停止/並行2スタックの
  拍整合/ぶつかり再発火条件。

### 3-5. 保存(draft)
- storage.jsに `studio: { works: [], draft: null }` を追加。SCHEMA_VERSION+1・
  デフォルト値マージ・書き出し/読み込みラウンドトリップ試験に含める
  (worksの中身UI は段階2だが、器は今回作る)。
- かきかけ自動保存: 編集操作のたびdraftへ(デバウンス可)。再入場時
  「かきかけの さくひんが あるよ」→続きから。保存ダイアログは作らない。

### 3-6. 受け入れ条件(段階1完了の定義)
- #studio-dev で3ペインが開き、キャラ2体にそれぞれプログラムを組んで▶で並行に動く。
- 18種全カードが仕様どおり動く(発光追跡・■初期化・全画面上演含む)。
- とりけし20手・長押しコピー・draft復帰が動く。
- 既存機能不変・verify全PASS・ラウンドトリップ試験PASS・エンジン単体テストPASS。
- 【停止】神田さんが実機確認(テンポ400ms・磁石・発光の最終判定)。
