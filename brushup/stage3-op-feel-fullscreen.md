## §1. 連続移動のカクつき・揺れ（不具合）

### 1-1. 原因（2つ重なっている・Chat 調査済み）
原因A: 移動の間隔とアニメーションの長さが噛み合っていない
- `.actor { transition: transform ${CFG.MOVE_MS}ms cubic-bezier(.45,.05,.55,1) }` ＝ 340ms かけて、ゆっくり出てゆっくり止まる（ease-in-out）
- 連続移動は `OP_MS = 100` ＝ 100ms ごとに次の移動を開始
- → 340ms のアニメーションが3割ほど進んだところで毎回中断・再スタート。「ゆっくり出る」部分だけが繰り返され、進んでは詰まる ＝ カクカク

原因B: 足踏み演出が100msごとに鳴り直している
- `@keyframes sbStep { 50% { transform: scaleY(.93) translateY(1.5px) } }` を `.actor .sp-in.stepA { animation: sbStep .34s ease-out }` で再生
- 1マス動くたびに走るので、340msの潰し演出が100msごとに再発火。潰れかけては戻るを繰り返す ＝ 小刻みに揺れる

### 1-2. 直し方（★1拍ごとの移動は今のまま）
連続移動（そうさ）のあいだだけ「歩き」として扱う。
- 移動アニメーションの長さを移動間隔と一致させ、イージングを等速（`linear`）にする。中断が起きなくなり、押しっぱなしのあいだ滑らかに進む。
- 足踏み演出（`sbStep`）を連続移動中は再発火させない。出したい場合は演出の長さも移動間隔に合わせること。340ms のまま再発火させると揺れが残る。
- `みぎへ すすむ` など1拍ごとの移動（`CFG.MOVE_MS = 340`・ease-in-out・`sbStep`）は一切変えない。ここは studio と共有で、b6a まで実機OKの手触り。そうさ由来の移動だけを別扱いにする。

`.actor` に「いま連続移動中」を示す状態（クラス等）を付け、そのときだけ transition と演出を切り替えるのが素直。実装方式は Code の判断でよいが、1拍ごとの移動に影響が出ていないことを必ず確認すること。

### 1-3. 移動の速さ（★基準モックからあえて外す・Chat 判断）
`OP_MS = 100` のまま維持する。
基準モック `dpad-play-mock.html` は `setInterval(step, 50)` ＝ 50ms間隔で、指示書は「モックの手触りが基準」としている。ここをあえて外す判断をした。理由:
- モックはトランジションなしの瞬間移動で、実装（340msのイージング付き）とは前提が違う。間隔の数字だけを移せば同じ手触りになる、という関係ではない
- 盤は12マス。50ms/マスだと端から端まで0.6秒。小1〜小2の指で押しっぱなしにすると確実に行き過ぎる
- 100ms/マスなら 1.2秒で横断。子ども向けゲームの移動速度としてはこちらが標準的
- そして 1-2 を直せば、100msでもモックの50msより滑らかに見える（等速の連続移動になるため）

`OP_MS` は名前付きの定数1つにまとめ、コメントに「実機の手触りで調整する値」と明記すること。実機で速さが合わなければ、この数字だけを変えて再調整する（速すぎ→130ms／遅すぎ→70ms が目安）。

## §2. 全画面（ひろげる）を1画面に収める

### 2-1. 現状
`.studio-root.big` が隠しているのは `.studio-pal` `.studio-asm` `.bgrow` `.castrow` `.rcap` `.editbtn`。残るのは プレビュー ＋ じゅうじキー ＋ ゲームのせってい。

### 2-2. ゲームのせっていを隠す
全画面は「遊ぶ」ためのモード。設定は要らない。`.studio-root.big` の非表示リストに `.gamecfg` を追加する。設定を変えたければ全画面を解除すればよい。

### 2-3. プレビューの高さの予備を取り直す（★マジックナンバーを増やさない）
現在 `.studio-root.big .theater { width: min(100%, calc((100dvh - 110px) * 1.5)) }`。この 110px は じゅうじキーが無かった頃の値。実測でじゅうじキーは約 148px（padding 12 ＋ 3行×42 ＋ 2×gap5）あり、足りていない。
- 単純に `110` を大きい数字に置き換えるのは不可。じゅうじキーは操作可能キャラがいる時だけ出るので、出ない場合に今度はプレビューが小さくなりすぎる。
- 満たすべき条件は次の2つ。実現方式は Code が選んでよい。
  1. じゅうじキーが必ず画面内に全部見える（縦スクロールなしで押せる）
  2. プレビューが画面からはみ出さず、余った高さを使ってできるだけ大きい
- じゅうじキーの有無で予備の高さが自動的に変わる作り（flex で残りを配分する等）が望ましい。新しい固定px値を足す解は避けること。

### 2-4. 通常画面（全画面でないとき）
じゅうじキーが増えたぶん、右カラムの縦がさらに厳しくなっている。ただし本便では通常画面の縦予算には手を入れない（「ゲームのせってい」再設計＝別便でまとめて組み直す）。通常画面でじゅうじキーが押せる位置に出ていることだけを確認すること。
